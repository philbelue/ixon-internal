06 · CONTEXT

Strategic interpretation

MiTek's surface ask is feature clarity. The deeper opportunity is multi-site hardware standardization with IXON as their secure remote-access / network-appliance layer.

Hardware-first protects the strategic priority

Devices in the ecosystem drive both customer value and IXON's WNC trajectory. The next 50–60 device wave is the most important thing to get unblocked. Feature clarity is a means; standardization is the end.

Identity governance is a deal-shaper, not an obstacle

Josh explicitly said IT now treats IXON as a network appliance. Entra/SCIM enforcement, vendor isolation, and least-privilege are not blockers to overcome — they are the language IT uses to certify IXON for enterprise scale. Lean into this. It is the rationale for standardization, not an obstruction to it.

Feature clarity converts to adoption velocity

MiTek does not need a feature inventory. They need to know which capabilities matter for their next rollout wave, which capabilities are already enabled but underused, and which capabilities to evaluate later. The v1 license summary should be structured this way (see section 16).

Friction risks are political risks

Earth City offline-device, VPN client inconsistency, and the branded-portal / Entra issue are operational items but each is also a credibility test with IT. Resolution speed signals scalability. Max should own the technical close-out path.

Premature SaaS / Edge App expansion is a trap

Edge App Management, API, session recording, access request, and Docker-based monitoring are all interesting to MiTek but none belongs in the upsell column until the rollout foundation (identity, group structure, hardware standardization) is locked. Selling them now risks both the foundation and the customer's confidence.

09 · ARCHITECTURE

Access governance model

Working architecture for MiTek's access governance — to be validated with Max and refined live with Josh and David. This is the foundation that must lock before broader feature/SaaS conversation is appropriate.

Recommended hierarchy

Group structure
  • Master group: MiTek production equipment
  • Subgroup layer 1: Facility (Missouri, Arizona, Czech, South Africa, UK, etc.)
  • Subgroup layer 2: Equipment type / product line (stamping presses, palletizers, builder products, etc.)
  • Group type remains internal vs. external (MiTek vs. outside vendor)
Role taxonomy (recommended)
  • Corporate admin — full visibility across MiTek-internal groups
  • Facility admin — scoped to a single facility
  • Vendor admin — scoped to a vendor group; can manage own users only
  • Vendor technician — scoped to a vendor group; operate-only
  • View-only — for audit/compliance roles
Access category design
  • Default "all devices" usage replaced with explicit access categories
  • Pending access request identified on March 6 to be the first least-privilege rebuild test case
  • Vendor access categories are equipment-type-scoped
Identity layer
  • Entra-enforced login on portal (configuration to be confirmed by Max)
  • SCIM for user lifecycle management
  • Manual role assignment until SCIM role replication is generally available Roadmap-dependent
  • External vendors: Entra B2B if MiTek prefers central identity control; otherwise vendor-admin self-service
Core principle — least privilege, vendor isolation

The group is the box. The role determines what can be done inside the box. Vendors should never see equipment outside their assigned scope. Internal MiTek teams should be scoped by facility and equipment type unless the role explicitly justifies broader visibility. This is the durable model regardless of which features get added later.
